# 第一部分：AI 壁垒方向卡

## 1.1 当前画像

没有计算机专业背景。接触代码是从"解决问题"开始的：手头有大量 PDF 需要提取数据、表格需要合并、PPT 需要从零生成。

技术栈：
- JavaScript/Node.js（够用来写文档处理脚本）
- 依赖包：`docx`（生成 Word）、`pdf-parse`（提取 PDF 文本）
- AI 工具：DeepSeek（主力）+ Claude Code CLI（任务编排）
- 工作场景：表格处理、PPT 制作、PDF 数据提取、文档自动化

## 1.2 AI 时代的差距认知

AI 时代真正的差距不在"会用 AI"和"不会用 AI"之间，而在**知道用 AI 解决什么问题的人**和**不知道问题在哪的人**之间。

## 1.3 选择的方向：文档智能处理

理由：这是我每天都在做的事情——PDF 提取、表格整理、Word 生成。问题高频、高耗、高错误率，非结构化到结构化的转换正是 AI 最擅长的。

## 1.4 壁垒定义

> 在"非结构化文档 → 结构化数据 → 格式化输出"这条链路上，建立端到端的 AI 自动化能力。

三层能力：
- 第一层（提取）：从 PDF/图片/扫描件中准确提取结构化数据
- 第二层（转换）：按业务逻辑整理、校验、清洗
- 第三层（输出）：生成符合格式要求的 Word/PPT/Excel

## 1.5 优劣势

**优势**：有真实场景、可快速验证、问题驱动、对准确率敏感
**短板**：缺少 CS 基础、AI 工具广度不够、英文阅读能力弱、缺少同行比较、停留在脚本层面

# 第二部分：AI 工具定位表

## 核心工具

| 工具 | 使用深度 | 解决的问题 |
|------|----------|------------|
| DeepSeek | 深度依赖 | 代码生成、文档理解、数据结构化 |
| Claude Code CLI | 深度依赖 | 多步骤任务编排、文件批量操作 |
| pdf-parse | 深度依赖 | PDF 文本提取（AI 处理起点） |
| docx | 深度依赖 | 代码生成 .docx 文件 |

## 扩展工具

| 工具 | 使用深度 | 定位 |
|------|----------|------|
| ChatGPT | 偶尔用 | 备选方案、交叉验证 |
| GitHub Copilot | 浅度体验 | IDE 代码补全 |
| 豆包 | 浅度体验 | 中文内容处理 |

## 当前工具链

```
PDF → pdf-parse → DeepSeek 结构化 → docx 生成 Word → 人工校验
              ↑ Claude Code CLI 贯穿全程
```

### 工具链问题
- 每步之间靠人工桥接，不是自动化管线
- OCR 缺失，图片型 PDF 卡住
- 没有模板系统，每次从零写样式
