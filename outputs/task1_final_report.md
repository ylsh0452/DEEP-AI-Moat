# AI 壁垒方向卡

---

## 1. 我为什么要学习 AI？

说实话，不是我自己主动要学的——是工作逼的。

我的日常就是处理各种文档：PDF 要提取数据、表格要合并、PPT 要从零生成。这些东西纯手工做又慢又容易出错。后来有人跟我说 DeepSeek 能帮忙写代码，我试了一下，发现确实能——我一个没系统学过编程的人，靠跟 AI 对话就能写出能用的脚本来处理文档。

所以原因其实很简单：**不学 AI，活就干不完。** 学会用它，同样的活花一半时间。不是什么宏大理想，就是实用。参加 DEEP 营也是因为这个——既然已经在用了，不如系统一点地学一下，看看别人是怎么用的，别自己闭门造车。

我没有想过"AI 时代会不会淘汰我"这种问题。对我来说，它就是当前手头工作的一个工具，跟 Excel 公式差不多——学好了能省很多力气。至于更深的东西，在 DEEP 营期间慢慢理解。

---

## 2. AI 未来可能在哪些方面改变学习、工作、比赛、项目或创业？

### 学习方面
AI 让"学不会"这件事的代价降低了。以前学一个新技术——比如 Node.js 的 `docx` 包——需要看文档、搜教程、自己试错，可能要花一个下午。现在我问 DeepSeek"怎么生成一个带表格边框的 Word 文档"，它直接给我能跑的代码。学习从"系统性啃教材"变成了"问题驱动的即时学习"——遇到什么问什么，边做边学。

### 工作方面
文档处理是我的主场景。AI 对工作的改变不是"做得更快"，而是**改变了能做什么**。以前：手动复制粘贴是从 PDF 提取数据的唯一方式。现在：pdf-parse + AI 结构化输出，十分钟搞定以前半天的工作。以后：整条管线自动化，人只需要最后确认。工作从"操作工"变成了"监督者"。

### 比赛/项目方面
AI 拉平了技术实现的入场门槛。以前做一个能用的工具需要前后端开发人员，现在一个人 + AI 就能搭出 MVP。这给个人和小团队参与比赛或做项目带来了巨大优势。但同时，当所有人都能用 AI 实现功能时，真正的差距就变成了**对问题的理解和解决思路**——而不是代码写得多好。

### 创业方面
我目前还没有明确的创业计划，但我能看到的趋势是：AI 驱动的垂直领域工具会是一个方向。不是做一个"通用的 AI 助手"，而是在某个具体场景（比如合同审查、发票录入、报表生成）里做到极致。这才是有护城河的。

---

## 3. 我现在和 AI 前沿使用者之间最大的差距是什么？

如果只许说一个：**我不知道"AI 能这样解决问题"的方法论。**

具体拆开，差距体现在三个层面：

- **工具选型差距**：我深度用 DeepSeek，但这不是因为我对比之后觉得它最合适——是因为我先接触到它，用习惯了。我不知道其他模型（Claude、Gemini）在我的文档处理场景里会不会更好。我缺少"根据任务选择最优模型"的判断力。

- **工作流差距**：我和 AI 的交互模式是"每次新开一个对话，描述需求，等回复，复制结果"。这是 L1 级别的使用方式。前沿使用者已经把 AI 嵌入了自动化 Pipeline——上游数据流入，AI 自动处理，结果自动输出。我还没到这个层次。

- **信息渠道差距**：我的 AI 信息源主要是实际操作中的自然发现，偶尔刷到的公众号文章。我没有系统地关注英文社区（Hacker News、Reddit AI 板块）、没有跟踪开源项目、没有订阅 Newsletter。这让我对"别人已经在做什么"缺乏感知。

一句话总结：**我停留在"手动操作 + AI 辅助"的模式，而没有进入"AI 驱动 + 人工监督"的模式。**

---

## 4. 我目前最需要解决的 3 个问题是什么？

### 问题一：PDF 到结构化数据的提取不稳定

当前用 pdf-parse 提取文本后丢给 AI 做结构化，但遇到以下情况就出问题：
- 扫描件 PDF（图片型）—— pdf-parse 提不出字，直接卡住
- 复杂嵌套表格—— AI 容易把表头和数据行混淆
- 中文竖排或混合排版——提取结果顺序错乱

**如果这个问题不解决，后续所有自动化都无从谈起。** 这是我的首要瓶颈。

### 问题二：没有可复用的 Pipeline，每次都是"从零开始"

我现在处理每个文档的流程是：手动开 AI 对话 → 描述任务 → 拿代码跑 → 跑不通改 → 再跑 → 拿到结果。下一个文档来了，同样的流程再走一遍。我没有积累 Prompt 模板，没有抽象出通用函数，没有形成一条"输入 PDF → 输出 Word"的标准管线。

这导致我的效率上限很低：处理得越多，边际成本降不下来。

### 问题三：缺少对标和参考系，不知道自己做到什么程度了

我不知道市面上有没有已经成熟的"文档 AI 处理"工具或开源项目。我不知道和我一样在做这件事的人用的是什么方案。我可能在用一把锤子敲螺丝——方向对了但工具不对。缺少外部参考让我对自己的进展无法做准确判断。

## 5. 我希望 AI 优先辅助我完成哪些具体任务？

按优先级排列：

| 优先级 | 任务 | 当前方式 | 期望方式 |
|--------|------|----------|----------|
| P0 | PDF 表格数据提取到 Excel | pdf-parse + AI 对话 + 手动整理 | 一条命令自动提取，输出结构化 JSON/CSV |
| P0 | 按模板生成 Word 报告 | 手动写 docx 代码 + AI 辅助 | 输入数据和模板名，自动生成 |
| P1 | 多来源数据合并与校验 | 手动对照、逐条比对 | AI 自动匹配合并 + 规则校验 + 异常标记 |
| P1 | PPT 大纲生成与内容填充 | 手动做 | 输入要点，AI 生成完整 PPT |
| P2 | 文档翻译与校对 | 逐段复制到 AI 翻译 | Pipeline 中自动调用翻译节点 |

这些不是什么宏大愿景——就是我在接下来三个月需要落地的东西。

## 6. 我未来想建立的个人 AI 壁垒是什么？

> **在"非结构化文档 → 结构化数据 → 格式化输出"这条链路上，建立端到端的 AI 自动化能力。不只是会调 API——而是能做到"别人做不了的准确度"和"别人做不了的效率"。**

展开来说，三层壁垒：

- **第一层（技术壁垒）**：能处理复杂文档。不只是文字型 PDF，还包括扫描件、手写体、印章遮挡、合并单元格表格。别人卡住的地方，我能打通。

- **第二层（效率壁垒）**：有可复用的 Pipeline 和 Prompt 模板库。不是每次从零开始，而是"接上数据源就跑"。别人的处理速度是"一天几份"，我能做到"一小时几十份"。

- **第三层（领域壁垒）**：对特定行业的文档格式有深入理解（合同、财报、政府公文等），能把行业规则编码到 AI 的校验逻辑里。通用 AI 做不到的准确率，我做得出来。

## 7. 如果别人也会使用 ChatGPT、Claude、Gemini、Codex，我准备如何和他们拉开差距？

这是一道灵魂拷问。坦率说，现在会用 AI 的人越来越多，光靠"我会用 ChatGPT"已经不算壁垒了。

拉开差距的方向不在"用什么工具"，而在以下三点：

### 7.1 不做"通用 AI 用户"，做"场景深度用户"

别人用 AI 聊天、写文案、查资料——什么都会用一点，什么都是浅浅一层。我专注一个场景：文档处理。我可以在这个场景里做到他们做不到的事——比如他们只会问 ChatGPT"帮我整理这个 PDF"，而我已经搭好了一条处理 Pipe，知道每种文档格式该用什么策略、Prompt 怎么写最稳、输出怎么校验。**深度就是对广度用户的最大壁垒。**

### 7.2 从"会用"到"会造"

大多数人停留在"打开 AI 对话框 → 输入问题 → 复制结果"。我要进到下一个阶段：把 AI 变成我搭的系统中的一个自动运行的组件。这需要我会写代码、会设计 Pipeline、会处理边界情况。**"会造"的人，和"会用"的人，差距会越来越大。**

### 7.3 用真实场景积累不可替代的经验

AI 生成的结果好不好，最终需要人来判断——而这个判断力来自真实经验。我每天都在处理真实的文档，知道哪些错误是行业特定的、哪些格式要求是甲方在意的。这种"领域判断力"是纯靠 AI 无法获得的。把 AI 能力和领域经验结合，才是真正难以复制的护城河。

---

## 8. 这个暑期结束时，我希望自己在哪些方面发生明显变化？

### 变化一：把 DeepSeek API 调通

这是最实际的——不要每次手动打开网页版粘贴文档、等回复、复制结果。改成写脚本调 API，至少跑通一个场景：输入 PDF → 输出结构化的 Excel。不需要多完美，能跑就行。

### 变化二：知道自己到底需要学什么

写这份文档之前，我对 AI 的认知就是"有个对话框，问它啥就答啥"。做完这个任务——特别是信息差观察那部分——我才知道有 OCR 模型、有 Function Calling、有一堆开源文档工具。暑假结束时，我希望不再是一问三不知的状态，至少对这些东西有个基本概念。

### 变化三：把日常的文档工作真正自动化掉一两项

不是"写了个脚本放着"，而是真的在日常工作中用起来。比如这周要处理的 PDF，不再手动做，用自己搭的流程跑一遍，看看效果。跑了才知道哪里有问题。

### 实事求是地说

我的目标不大。暑假结束时我不会变成什么"AI 专家"，也不会有多完善的 Pipeline。但我希望至少做到：**遇到一个文档，第一反应不再是自己手动处理，而是想想能不能用 AI 自动搞定。** 思维的转变比技术更重要。

---

## 总结

我没有计算机背景，我不是在追赶 AI 的技术前沿。我要做的是：**在自己每天面对的真实文档工作场景里，用 AI 建立别人追不上的效率壁垒。** 这个壁垒不是靠比别人更会用 ChatGPT 建立的——而是靠比别人更懂自己的场景、更快地把 AI 嵌入工作流、更持续地积累领域经验建立的。

这个夏天，我要迈出第一步：从"聊天式用 AI"走到"管线式用 AI"。

## 五、工具选择原则

1. **场景优先，不追新品**：不是因为某个工具热门就去学，而是因为"我的场景需要它解决什么问题"
2. **先用熟再换**：DeepSeek 和 Claude Code 已经深度使用，先挖透它们的能力（Function Calling、Workflows），再考虑引入新工具
3. **对比实测，不盲信宣传**：同一个文档，两个工具分别跑，看数据而不是看宣传

### 误区五：把 AI 当答案，不把它当工具
**表现**：AI 说这个方案最好，我就信。AI 说这个代码没问题，我就不测。
**我的应对**：AI 是技术杠杆，不是技术裁判。**最终的判断力必须在我自己手里——这是整个路线图最核心的一条原则。**

## 一、完成这个任务后，我对 AI 的理解发生了什么变化？

### 写之前我的理解

写这份文档之前，我对 AI 的理解就是：有个聊天窗口，问什么答什么。DeepSeek 帮我写代码、理解 PDF、生成文档——反正遇到问题打开窗口打字就行。没想过"我在哪个层次"、"别人用到了什么程度"这些问题，就是当工具用。

### 写之后的变化

说实话，变化不是那种"豁然开朗"的感觉——更多是"原来我落后了这么多"。

**变化一：知道了很多之前不知道的东西**

DeepSeek 有 API、有 Function Calling、有专门的 OCR 模型。有一堆开源的文档 AI 工具（MinerU、Docling、pdf-mcp）。Claude Code 能做多 Agent 编排。**这些东西写文档之前我一个都不知道。** 不是它们不存在，是我从来没去了解过。

**变化二：发现"会用"和"会造"之间的距离**

我每天用 DeepSeek 和 Claude Code，觉得自己算是"会用 AI"的人了。但写完这份文档发现——我只是在"手动用"，离"程序化调用"、"自动管线"还差得远。就像会用 Word 打字和会用 VBA 写宏的差距。

**变化三：被逼着想了一些原本不会想的问题**

如果没有这个任务，我不会去想"我未来三年要建什么壁垒"、"别人也会用 AI 我怎么拉开差距"这种问题。写的时候有些问题回答起来挺吃力的，因为没有现成的答案。但回答完之后，确实比之前清楚了一些。

### 一句话总结

> **之前觉得自己"会用 AI"，写完之后发现自己只是"用到了 AI 最基础的功能"，而且落后了外面至少半年。不过至少现在知道自己落后在哪了，比之前"不知道落后"是进步。**

## 二、我是否发现自己之前存在工具焦虑、信息焦虑或伪需求问题？

### 工具焦虑：之前没有，写的过程中出现了

在写这份文档之前，我没什么工具焦虑——因为我只用 DeepSeek 和 Claude Code，也不了解其他工具。焦虑是在写信息差观察时出现的：一下子发现了五个开源工具，每个看起来都很好，我该学哪个？MinerU 是 Python 的，我要不要转语言？pdf-mcp 看起来很新，敢不敢踩坑？

**诊断**：这不是"真焦虑"，而是"一下子暴露了太多信息"。应对方式很简单：**不急着全学。** 先对比测试两个（MinerU vs DeepSeek-OCR-2），哪个好用用哪个。其他的保持关注，但不动手。

### 信息焦虑：一直存在，现在更清晰了

信息焦虑是我最大的问题，但写之前我说不清楚。现在我能说清楚了：**我的焦虑不是因为信息太少，而是因为信息渠道太随机。** 我获取 AI 信息的方式是"碰到了就碰到了，碰不到就算了"。这种随机性让我对自己的判断没有信心——我不知道自己是不是在用一种已经过时的方法做事。

**诊断**：解决方式不是"看更多"，而是"建立固定渠道 + 固定频率"。每周一次 GitHub Trending + Hacker News，就这两个渠道，固定时间看，不看多。

### 伪需求：认真想了一下，确实有一些

整理工具定位表时，我发现自己列了几个"想学但理由不充分"的工具：Cursor（因为大家都说好，但我其实对 Claude Code CLI 很满意）、LangChain（因为很多人提，但我 Node.js 直调 API 够用）、Midjourney（跟文档处理没什么关系）。

**诊断**：这些是伪需求——不是因为我的场景需要它们，而是因为"别人在用"或"听起来不错"。处理方式：从待学列表中删掉，除非某个具体场景明确需要，否则不花时间。

### 最想铲除的一个伪需求

**"我需要系统学 Python 才能做文档 AI。"** ——写这份文档之前，我一直觉得 Python 是文档 AI 的入场券，因为大部分工具是 Python 写的。但我现在想清楚了：我不需要成为一个 Python 开发者，我只需要能跑通一个 Python 脚本。DeepSeek 可以帮我写 Python 代码，我的任务是判断它跑得对不对。**用 Node.js 做主力开发 + AI 帮我处理 Python 部分，比花三个月系统学 Python 更高效。**

## 三、我现在最应该优先学习哪 1–2 个 AI 工具？

### 成本收益分析

我把所有"想学"的工具放在一起，用两个维度做了排序：

| 工具 | 学习成本 | 预期收益 | 优先级 | 理由 |
|------|----------|----------|--------|------|
| DeepSeek API + Function Calling | 低（我已经会用网页版，API 文档花半天能看完） | 极高（把"手动聊天"变成"程序化调用"，是整个 Pipeline 的基础） | 🔴 **第一优先** | 不学会这个，Pipeline 无从谈起 |
| DeepSeek-OCR-2 或 MinerU | 中（需要对比测试、可能涉及 Python 环境配置） | 极高（填补当前工具链最大的 OCR 缺口） | 🔴 **第二优先** | 扫描件 PDF 目前完全无解，这是硬瓶颈 |
| Claude Code Dynamic Workflows | 中（需要学 Workflow 脚本语法） | 高（替代手写 Node.js 流程控制，快速验证 Pipeline 可行性） | 🟡 | 前两个搞定后再学，不着急 |
| Python 基础 | 高（需要系统学习） | 中（只是为了让其他工具跑起来，不是独立目标） | ⬇️ 降级 | 按需学习，不系统学 |
| pdf-mcp / ParseHawk | 低 | 中（锦上添花，不是雪中送炭） | ⬇️ 降级 | 等主链路跑通后再看 |

### 结论：这个夏天只死磕两个

1. **DeepSeek API + Function Calling**——优先级最高，7 月内必须接入并跑通第一个自动化脚本
2. **一个 OCR 方案**（DeepSeek-OCR-2 或 MinerU）——7-8 月对比测试，选一个集成

**这两个搞定之前，其他工具一律保持关注但不动手。** 这是我从"工具焦虑"里爬出来的方式：不是全学，而是只学最关键的。

## 四、我目前最薄弱的壁垒是什么？

### 排个序

| 薄弱环节 | 严重程度 | 具体表现 |
|----------|----------|----------|
| **信息获取** | 🔴 最弱 | 不知道外面在发生什么，用了几个月的 pdf-parse 不知道有更好的方案 |
| **Pipeline 设计** | 🔴 最弱 | 还在"手动聊天"模式，没有自动化管线 |
| **英文阅读** | 🟡 弱 | 大量一手资料是英文的，阅读速度慢导致信息滞后 |
| **产品化思维** | 🟡 弱 | 只给自己写工具，没想过"别人也能用" |
| **代码架构** | 🟡 弱 | 脚本能跑但结构不好，需要 AI 大量辅助 |

### 最薄弱的分析

**信息获取是第一薄弱环节。** 它不只是"获取信息慢"的问题——它是所有其他短板的放大器。如果我早三个月知道 MinerU 或 pdf-mcp 的存在，我的 PDF 处理方案可能早就是另一个水平了。信息差导致的损失不是"慢了三个月"，而是"三个月的时间花在了重复造轮子上"。

**Pipeline 设计是第二薄弱环节。** 但这个问题和信息差直接相关——如果我早知道 Claude Code 有 Dynamic Workflows，我的管线设计思路可能完全不同。所以第一和第二是因果关系：**先补信息差，信息足了，Pipeline 的设计就会有更好的输入。**

### 短期对策

- **信息差**：每周日晚上花 30 分钟刷 GitHub Trending + Hacker News，建一个"文档 AI"关注文件夹
- **Pipeline**：在 Claude Code 里用最简单的方式先跑通一条链路（哪怕只是把 pdf-parse + DeepSeek API + docx 用 Workflow 脚本串起来），不追求完美，先追求"有一条能跑的"

## 五、下一步我准备如何通过任务二进入真实场景应用？

### 我的理解

任务二的标题是"真实场景应用"。结合我这份壁垒方向卡的结论，任务二对我而言不是"再写一份文档"，而是**"把我文档里写的东西在真实场景里跑起来"**。

### 具体计划

**Step 1：选一个最小的真实场景（本周）**

从我每天处理的文档里挑一个最典型的任务——比如"把一份 20 页的 PDF 报告里的表格数据提取到 Excel"。这个任务我有真实需求、有对比基线（我知道纯手工做要多久）、有判断标准（提取准确率是多少）。

**Step 2：用 DeepSeek API 替代手动聊天（下周）**

现在的流程：手动打开 DeepSeek 网页版 → 粘贴 PDF 文本 → 描述需求 → 复制结果 → 手动整理。任务二要实现的流程：一个 Node.js 脚本，读取 PDF → 调 DeepSeek API with Function Calling → 自动输出结构化 JSON。关键是：**全程不需要我手动跟 AI 对话。**

**Step 3：对比数据，记录差距（过程中）**

同一份 PDF，记录三组数据：
- 纯手工处理耗时 vs 自动化脚本耗时
- 手工提取准确率 vs AI 提取准确率
- 哪些环节 AI 做得好、哪些需要人工修正

这些数据就是任务二的交付物核心——不靠"我觉得"，靠"数据说"。

**Step 4：产出任务二文档（最后）**

把以上的场景选择、实现过程、数据对比、踩坑记录整理成任务二的交付文档。格式不重要——真实的数据和过程才重要。

### 任务二的"完成标准"（我自己定的，不等教练定）

- ✅ 有一个真实 PDF 文件（不是 Demo，是我工作中真正要处理的）
- ✅ 有一条命令能从 PDF 输出结构化 JSON
- ✅ 有对比数据（手工 vs AI，耗时和准确率）
- ✅ 知道哪个环节最需要改进
- ❌ 不需要 Pipeline 完美——v0.1 就可以，先跑通

### 与本次任务一的衔接

任务一让我知道了：
1. 我当前在"手动用 AI"阶段，需要进到"管线用 AI"
2. 我的信息渠道太窄，已经有更好的工具我不知道
3. 我的方向是文档智能处理，OCR 是第一个要补的缺口

任务二要验证的是：
1. DeepSeek API 能不能真的替代手动聊天（技术验证）
2. 自动化处理能比纯手工快多少、准多少（效率验证）
3. 我在真实场景中的判断力够不够用（能力验证）
