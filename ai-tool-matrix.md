# AI 工具定位表

## 说明

下表记录了我在实际工作中接触过的 AI 工具。每个工具都标注了：
- **使用深度**：偶尔用 / 日常用 / 深度依赖
- **解决的问题**：具体场景，不是泛泛的"提高效率"
- **局限性**：我在实际使用中遇到的真实问题

## 想探索但还没深入的工具

| 工具 | 想解决的问题 | 为什么还没用 |
|------|-------------|-------------|
| **OCR 工具**（如 PaddleOCR） | 扫描件 PDF / 图片中的文字识别 | 技术门槛比纯文本提取高，需要先解决部署和环境问题 |
| **LangChain** | 把 AI 调用做成可编排的 Pipeline | 担心引入过多抽象层，目前 Node.js 直连 API 够用 |

| **Midjourney / DALL-E** | PPT 配图生成 | 与当前文档处理的文本型工作关联度不高 |

## 工具组合的现状

```
当前工作流：
PDF 文件 → [pdf-parse 提取文本] → [人工整理或脚本预处理]
→ [DeepSeek 理解内容+输出结构化数据] → [Node.js docx 生成 Word]
→ [人工最后校验]

Claude Code CLI 贯穿全程：写脚本、调试、改代码
```

### 当前工具链的问题
1. **太依赖人工桥接**：每一步之间需要我手动操作，不是自动化管线
2. **没有统一入口**：PDF 提取用一个脚本，AI 处理用一个对话，文档生成又用一个脚本
3. **OCR 缺失**：遇到图片型 PDF 就直接卡住
4. **没有模板系统**：每次生成文档都是"从零写样式"，重复劳动

### 工具链改进方向
- 把 pdf-parse → DeepSeek API → docx 串成一条 CLI 命令
- 建立常用的文档模板（报告、合同摘要、数据简报）
- 接入 OCR 能力覆盖图片型文档
